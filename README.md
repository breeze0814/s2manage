# S2A Manager

S2A Manager 是一个面向 Sub2API 站长的运维管理面板，用于集中管理 Sub2API 站点的分组、账号、倍率、上游可用性、日志和自动化同步任务。

它不是 Sub2API 本体的替代品，而是 Sub2API 管理后台的增强工具。项目重点解决站点长期运营中常见的维护问题：分组倍率频繁调整、上游账号状态不稳定、采集源倍率变化、分组或账号删除重建后绑定失效、日志难读和异常数据残留等。

项目仓库：[langrenjh-alt/S2A-Manager](https://github.com/langrenjh-alt/S2A-Manager)

如果你需要一个现成可用的 SUB2API 中转站，可以访问 [https://z30.top](https://z30.top)。z30.top 可作为中转服务入口，也适合作为 S2A Manager 连接、倍率同步和站点运营流程的参考站点。

## 适合场景

- 站点中有较多分组，需要批量维护或自动同步分组倍率。
- 有多个上游账号，需要持续检测账号是否可用。
- 需要从外部采集源站点同步分组倍率，并按规则应用到目标 Sub2API。
- 经常遇到分组删除、重建、改名后绑定关系失效的问题。
- 希望集中查看服务状态、日志、账号余额、异常数据和自动化任务结果。
- 希望减少人工排查、手动改倍率和清理脏数据的工作量。

## 主要功能

### 多连接管理

- 支持维护多个 Sub2API 管理连接。
- 支持连接启停、连接测试和服务状态查看。
- 支持 Web 页面和后台 worker 配合运行。

### 分组与倍率规则

- 查看和维护目标 Sub2API 分组。
- 为分组配置采集源绑定。
- 支持按首个源倍率、平均值、最低值、最高值或自定义公式计算目标倍率。
- 支持手动应用倍率规则，也支持 worker 自动应用。
- 当目标分组被删除或重建导致 ID 失效时，会自动清理无效绑定和规则，减少重复报错。
- 当分组绑定的采集源站被删除或源分组失效时，会移除对应来源绑定，并禁用已经没有可用来源的自动倍率规则。

### 账号调度管理

- 新增、编辑、删除账号。
- 启用或停用账号调度。
- 刷新账号凭证、清除账号错误、测试账号模型可用性。
- 为账号配置采集源绑定和账号倍率规则。
- 展示账号余额，并支持为单个账号设置余额预警阈值。
- 当账号余额低于阈值时，在页面顶部提示充值，并将余额高亮显示。
- New API 站点余额暂不参与余额预警，避免不可靠余额数据造成误报。

### 倍率采集与同步

- 在站内维护采集源站点。
- 定时采集源站分组倍率、模型价格和倍率变化。
- 支持按采集源充值倍率换算实际倍率。
- 当源站分组名称变化时，会同步更新本地绑定显示名称。
- 当源站分组删除、重建，或采集源站被删除后，会清理失效来源绑定，避免继续使用过期数据。
- 支持采集完成后自动应用已启用的分组或账号倍率规则。

### 上游账号检测

- 为账号配置上游可用性检测规则。
- 支持定时检测指定模型。
- 连续失败达到阈值后，可自动暂停账号调度。
- 检测恢复后，可自动或手动恢复账号调度。
- 账号暂停期间，可临时排除该账号绑定的采集源分组参与分组倍率计算；恢复调度、删除绑定、删除源站或删除目标分组/账号时，会同步清理相关排除记录。
- 支持设置检测超时、并发和 worker 执行间隔。

### 服务状态与无效数据清理

- 查看 Web、数据库、worker 心跳、采集源、连接和最近任务日志状态。
- 提供“清理无效数据”按钮，用于处理目标分组或账号被删除、重建后 ID 变化、采集源站被删除，以及连接配置或密钥异常导致的残留数据。
- 清理范围包括无效采集源绑定、倍率规则、公告目标分组、上游检测规则、上游暂停倍率源排除记录等本项目维护的数据。
- 清理结果会展示失效源绑定、失效暂停源、禁用规则等统计，方便确认历史残留是否已处理。

### 日志管理

- 支持按连接、级别、状态、动作、目标、时间和关键词筛选日志。
- 支持日志保留时间配置、清理过期日志和清空日志。
- 日志详情会将常见 JSON 字段转换为中文说明。
- 目标列会优先显示中文对象，例如“分组：xxx”“账号：xxx”“连接：xxx”，不直接暴露 `group:22`、`account:2113` 这类内部标识。

### 公告与站点设置

- 支持维护 Sub2API 常用站点设置。
- 支持倍率变化公告规则。
- 当目标分组倍率实际发生变化时，可自动发布公告。

## 技术栈

- Next.js 14 / React 18
- tRPC / TanStack Query
- Prisma 5 / PostgreSQL
- Tailwind CSS
- 后台 worker：`tsx src/worker/monitor.ts`

## 快速开始

项目使用 PostgreSQL。

```bash
npm ci
cp .env.example .env
npx prisma migrate deploy
npm run dev
```

首次访问 `http://127.0.0.1:3000` 后创建管理员账号。

后台自动任务需要单独启动：

```bash
npm run worker
```

worker 负责：

- BL 分组倍率自动采集和同步。
- 已启用分组或账号倍率规则自动应用。
- 上游账号检测、自动暂停和自动恢复。
- 倍率变化公告规则自动发布。

默认 worker 每 10 分钟运行一轮，可在右上角“应用设置”里调整。最小间隔为 1 分钟。`S2A_WORKER_INTERVAL_SECONDS` 只作为数据库未配置时的启动默认值。设置 `S2A_WORKER_ONCE=1` 可只运行一轮后退出，便于排查问题。

## 环境变量

`.env.example` 提供最小配置：

```env
DATABASE_URL="postgresql://s2amanager:password@127.0.0.1:5432/s2amanager?schema=public"
APP_SECRET="change-me-to-a-24-plus-char-secret!"
ENCRYPTION_KEY="AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA="
```

生产环境必须替换：

- `APP_SECRET`：至少 24 位随机字符串。
- `ENCRYPTION_KEY`：32 字节 base64 字符串，可用 `openssl rand -base64 32` 生成。已有加密数据后不要更换。
- `DATABASE_URL`：PostgreSQL 连接字符串，例如 `postgresql://USER:PASSWORD@HOST:5432/DB?schema=public`。

可选环境变量：

- `S2A_LOG_DIR`：运行日志目录，默认是项目目录下的 `logs/`。
- `S2A_WORKER_INTERVAL_SECONDS`：数据库未配置 worker 间隔时的启动默认值。
- `S2A_WORKER_ONCE=1`：让 worker 只执行一轮后退出。
- `S2A_UPSTREAM_MONITOR_TIMEOUT_SECONDS`：数据库未配置检测超时时的启动默认值。
- `S2A_UPSTREAM_MONITOR_CONCURRENCY`：数据库未配置检测并发时的启动默认值。

## 部署

Ubuntu 部署请看 [部署指南.md](./部署指南.md)。

推荐在服务器上单独维护 `.env`，不要把生产密钥、数据库连接、日志、构建产物、依赖目录或压缩备份包提交到仓库。

基本部署流程：

```bash
npm ci
npx prisma migrate deploy
npm run build
npm run start
```

生产环境需要同时托管 Web 服务和 worker。可以使用 systemd、pm2 或其他进程管理工具分别运行：

```bash
npm run start
npm run worker
```

## 注意事项

- Web 服务和 worker 是两个进程，生产环境需要分别托管。
- 倍率采集源在“倍率采集”页面中维护，不需要额外配置独立同步服务。
- 删除倍率采集源时会同步清理指向该源的分组/账号绑定和上游暂停倍率源排除记录；历史残留可在“服务状态”页面执行“清理无效数据”处理。
- 分组或账号倍率规则只有开启“使用倍率规则”后才会自动应用。
- 公告自动发布只会在目标分组倍率实际变化时触发。
- 如果上游检测规则设置为 1 分钟，worker 运行间隔也建议设置为 1 分钟。
- 如果单次检测超时或账号数量较多导致一轮耗时超过 worker 间隔，worker 会在本轮结束后继续补跑下一轮，但不会并发重叠执行。
- 中文内容按 UTF-8 发送；如果远端公告仍乱码，需要检查反向代理和 Sub2API 站点的编码处理。
