# S2A Manager Ubuntu 部署指南

本文档面向 Ubuntu 服务器部署。项目需要 Node.js 20+、npm、PostgreSQL 数据库、可访问 Sub2API 管理接口的网络环境，以及一个持久化目录保存本地运行日志。

## 1. 上传并解压

```bash
mkdir -p /opt/s2a-manager
cd /opt/s2a-manager
unzip S2A-Manager-ubuntu-deploy.zip
```

如果压缩包解压出 `S2A-Manager/` 子目录，后续命令请先进入该目录。

## 2. 安装依赖

```bash
npm ci
```

## 3. 配置环境变量

```bash
cp .env.example .env
mkdir -p /opt/s2a-manager/data
mkdir -p /opt/s2a-manager/logs
openssl rand -base64 32
```

编辑 `.env`：

```env
DATABASE_URL="postgresql://s2amanager:password@127.0.0.1:5432/s2amanager?schema=public"
APP_SECRET="请替换为至少24位的随机字符串"
ENCRYPTION_KEY="请替换为 openssl rand -base64 32 生成的值"
```

注意：
- `APP_SECRET` 用于登录会话签名，生产环境必须设置。
- `ENCRYPTION_KEY` 必须是 32 字节 base64 字符串，已有数据加密后不要再更换，否则已保存的 API Key 无法解密。
- `DATABASE_URL` 必须是 PostgreSQL 连接字符串。业务运行日志写入 `logs/s2a-manager-YYYY-MM-DD.log`，不再写入数据库。
- `.env` 不应提交或公开。

## 4. 初始化数据库并构建

全新部署时执行：

```bash
npx prisma migrate deploy
npm run build
```

如果这是从旧版 SQLite 生产库升级到 PostgreSQL，请先停止部署流程。`npx prisma migrate deploy` 只会在 PostgreSQL 中创建当前版本表结构，不会自动迁移 SQLite 中已有的管理员账号、连接配置、分组规则、采集源、公告规则等业务数据。需要先备份 SQLite 数据库文件，并单独准备和验证 SQLite 到 PostgreSQL 的数据迁移方案；确认数据已迁入 PostgreSQL 后再继续构建和启动。

## 5. 启动 Web 服务

```bash
npm run start
```

默认监听 `http://127.0.0.1:3000`。首次访问会进入初始化页面，创建管理员账号。

如果前面接 Nginx、宝塔反向代理、CDN 或其他缓存层，请不要缓存后台页面。建议至少保留下面的响应头，不要把 `/`、`/login`、`/setup` 或 `/_next/data` 一类页面代理缓存给其他用户：

```nginx
proxy_set_header Host $host;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
proxy_set_header X-Forwarded-Proto $scheme;
add_header Cache-Control "no-store, no-cache, must-revalidate, proxy-revalidate" always;
```

本项目已在服务端和 middleware 层校验登录态，未登录访问后台会重定向到 `/login`。如果升级后仍看到旧后台页面，请先清理浏览器缓存，并确认反向代理没有缓存旧版 HTML。

## 6. 启动后台 worker

worker 负责 BL 分组倍率同步、上游源站检测、暂停调度后的持续探活与自动恢复、自动公告规则发布等后台任务。Web 服务启动后，还需要单独运行：

```bash
npm run worker
```

默认 worker 是常驻进程，每 10 分钟运行一轮，并把心跳写入数据库。管理台的“服务状态”页面会根据心跳判断 worker 是否在线。运行间隔、上游检测超时和检测并发可以在右上角“应用设置”里调整；`S2A_WORKER_INTERVAL_SECONDS` 只是数据库未配置时的默认值。

上游检测达到连续失败阈值后，只会暂停该账号参与 Sub2API 调度，不会停止监测。暂停期间仍会按该规则的检测间隔继续探活；一旦检测成功，worker 会立即恢复该账号调度。即使一直未恢复，暂停时间到达后也会自动尝试恢复。

生产环境建议用 systemd 托管 Web 和 worker。

### Web systemd 示例

创建 `/etc/systemd/system/s2a-manager-web.service`：

```ini
[Unit]
Description=S2A Manager Web
After=network.target

[Service]
Type=simple
WorkingDirectory=/opt/s2a-manager
Environment=NODE_ENV=production
ExecStart=/usr/bin/npm run start
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

### Worker systemd 示例

创建 `/etc/systemd/system/s2a-manager-worker.service`：

```ini
[Unit]
Description=S2A Manager Worker
After=network.target

[Service]
Type=simple
WorkingDirectory=/opt/s2a-manager
Environment=NODE_ENV=production
Environment=S2A_WORKER_INTERVAL_SECONDS=600
Environment=S2A_UPSTREAM_MONITOR_TIMEOUT_SECONDS=45
Environment=S2A_UPSTREAM_MONITOR_CONCURRENCY=3
ExecStart=/usr/bin/npm run worker
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

启用服务：

```bash
systemctl daemon-reload
systemctl enable --now s2a-manager-web
systemctl enable --now s2a-manager-worker
systemctl status s2a-manager-web
systemctl status s2a-manager-worker
```

## 7. 升级部署

如果当前生产环境已经使用 PostgreSQL，升级新版本时：

```bash
systemctl stop s2a-manager-web s2a-manager-worker
cp -a /opt/s2a-manager /opt/s2a-manager.backup.$(date +%Y%m%d%H%M%S)
unzip -o S2A-Manager-ubuntu-deploy.zip -d /opt/s2a-manager
cd /opt/s2a-manager
npm ci
npx prisma migrate deploy
npm run build
systemctl start s2a-manager-web s2a-manager-worker
```

不要覆盖或删除 `.env` 和 `logs/` 目录。数据库数据保存在 PostgreSQL 中，升级前请按你的数据库运维策略做好备份。

如果当前生产环境仍是旧版 SQLite，不要直接执行上面的升级命令。请先停止服务、备份 SQLite 数据库文件和 `.env`，准备 SQLite 到 PostgreSQL 的数据迁移方案，并在测试库验证管理员账号、连接、分组规则、采集源、公告规则等数据完整后，再切换生产 `DATABASE_URL` 到 PostgreSQL 并执行 PostgreSQL 版部署。

## 8. 常见检查

```bash
journalctl -u s2a-manager-web -f
journalctl -u s2a-manager-worker -f
curl -I http://127.0.0.1:3000
```

如果中文公告发送后出现乱码，请确认服务器、反向代理和 Sub2API 站点都按 UTF-8 处理请求；本项目发送公告时会使用 UTF-8 JSON 请求体。
